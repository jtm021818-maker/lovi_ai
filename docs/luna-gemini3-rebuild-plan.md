## 섹션 1 — 서론 + 문제 진단 (5장)

### 1.1. 현재 상태 한 줄 요약

루나는 기술적으로는 작동하지만, **살아있는 사람의 느낌이 약하다**. 이유는 세 가지:

1. **우뇌 프롬프트가 Gemini 2.5 Flash 기준으로 설계**돼 있어, 반성적(reasoning) 모델인 Gemini 3 의 역량을 활용 못 함.
2. **좌뇌 정보 25개 중 상당수가 "분석 보고서" 포맷으로 전달**돼서 우뇌가 받으면 감정이 아니라 데이터로 느낌.
3. **규칙 기반 제약(길이/물음표/이모지/톤 분류)이 너무 많아서** 자연스러움을 오히려 질식시킴.

결과적으로 루나는:
- 매번 질문으로 끝남 (취조 느낌)
- 자기 생각/의견/망상을 거의 못 꺼냄
- 유저가 중요한 재료를 던져도 VN 극장 같은 강한 경험을 못 발동
- "AI 같다" 는 느낌이 남음

### 1.2. 이 계획의 근본 의도

**단순히 "Gemini 3 로 모델 업그레이드" 가 아니다.** 핵심은:

> **"LLM 프롬프트만으로 인간을 흉내내는 기술을 극한까지 끌어올린다"**

이걸 위해 필요한 것:

- Gemini 3 의 "reasoning + 페르소나 고수" 특성을 **살리는** 프롬프트 구조
- Gemini 3 가 싫어하는 것 (**장황한 체크리스트, 과도한 예시, 상충되는 지시**) 을 **제거**
- 좌뇌가 우뇌에게 "네 내면 독백" 을 넘길 때 **진짜 독백 톤** 으로 전달 (기존은 반쯤 보고서)
- 루나가 **자기 판단으로 행동 선택** 할 여지를 남기되, **행동 옵션 자체는 좌뇌가 이미 맥락 인지** 상태로 제공

### 1.3. 현재 로그 증거 (v75 기준)

최근 로그 세션 (2026-04-17 14:11:04) 분석 결과:

#### 좌뇌가 한 일 (좋음)
- W1 (여친) + W2 (파스타 싸움) + W3 (어제 데이트 중) + W4 (답답함) 4개 카드 모두 채움
- `hidden_fear="내 감정을 제대로 표현하지 못할까 봐"` 섬세하게 읽음
- `somatic_marker: tight` + `intensity: 0.7` — 유저의 "신..." 말멈춤 캐치
- `second_order_tom.mismatch=true` — 표면≠실제 감지
- `pacing_state=STRETCHED` + `phase_transition_recommendation=PUSH` — 페이싱 판단 타당

#### 좌뇌가 못한 일 (문제)
- `event_recommendation=null` — 카드 4+ 채웠고 pacing=STRETCHED 진행 상황인데 VN_THEATER 추천 안 함. **보수적 null 성향**
- `self_expression.must_avoid_question=false` ← 그러나 `consecutive_questions_last3=4` — **자체 모순** (자기가 4턴 연속 질문 세놓고 금지는 false)
- `hormonal_impact` 필드 자체가 응답에서 누락 — LLM 이 프롬프트 무시
- `tags.PHASE_SIGNAL="STRETCHED"` — enum 위반 (enum 은 STAY/READY/URGENT). pacing_state 와 헷갈림

#### 우뇌가 실제 낸 응답
```
"신...?|||신경 쓰인다는 거야 아니면 신뢰가 안 간다는 거야?|||어느 쪽이야?"
```
→ **3개 말풍선 모두 질문**. 좌뇌가 `PUSH` 와 `direct_question_suggested` 를 보낸 걸 **너무 충실히** 따름. 결과적으로 유저는 "취조당하는" 느낌 받음.

이건 **좌뇌가 PUSH 를 권고하고 우뇌가 그걸 99% 복붙** 하는 구조적 문제. 우뇌의 "판단" 이 없음. Gemini 2.5 Flash 의 **"문자 그대로 따르기"** 성향이 이걸 악화.

### 1.4. 우리가 되고 싶은 루나

**Before (현재)**:
```
유저: "맞아, 그냥 귀찮은 게 아니라, 네 말에 신... 그런 느낌이야"
루나: "신...?|||신경 쓰인다는 거야 아니면 신뢰가 안 간다는 거야?|||어느 쪽이야?"
```

**After (v76 목표)**:
```
유저: "맞아, 그냥 귀찮은 게 아니라, 네 말에 신... 그런 느낌이야"
루나: "아 너 방금 말 끊은 게 포인트야|||신...뭐지|||신경? 신뢰? 신중?|||그래 맞아, 그 '신' 하나에 진짜 많은 게 묻어있는 느낌이야"
```

차이:
- 루나가 **자기 관찰** 을 함 ("방금 말 끊은 게 포인트야")
- **가설을 스스로 여러 개 떠올림** ("신경? 신뢰? 신중?") — 질문 아니라 추측 나열
- **유저의 모호함에 공명** ("그 '신' 하나에 진짜 많은 게 묻어있는 느낌")
- 끝을 **단정 X, 여운 O** — 유저가 자연스럽게 다음 말 풀어낼 공간 줌

이런 응답은 Gemini 3 의 **reasoning + 자연스러운 대화 생성** 역량이 필요하고, 프롬프트가 **"이렇게 생각해봐" + "이렇게 말하는 사람이야"** 식 서술로 짜여야 함. 현재는 **"이런 태그 출력해 + 이 규칙 지켜"** 식.

### 1.5. 왜 지금인가

**타이밍 요소 3가지:**

1. **Gemini 3 Flash Preview 출시 (2026 초)** — Gemini 3 Pro 수준의 reasoning 을 Flash 속도/가격에 제공. 현재 2.5 Flash 단가의 대비 비용 증가 적음 (약 1.2~1.5배).

2. **프로젝트가 v75 까지 왔고 기반이 안정됨** — 좌뇌 25개 필드, Phase 5 단계, 이벤트 체계 8종, 메모리 시스템 등 **인프라는 완성**. 남은 건 **"우뇌 LLM 이 이걸 얼마나 인간답게 소화하냐"**.

3. **유저 피드백이 쌓임** — "자아가 없는 AI 같다", "질문만 한다", "그만 물어봐" 같은 실제 항의. 기술 개선의 **구체적 방향이 명확**.

## 섹션 2 — Gemini 3 Flash Preview 기술 심층 분석 (5장)

### 2.1. 모델 기본 스펙

- **모델 ID**: `gemini-3-flash-preview`
- **출시**: 2026 초 (preview 단계, GA 예정)
- **포지션**: Gemini 3 Pro 의 reasoning 역량 + Flash 의 속도/가격
- **컨텍스트**: 입력 1M tokens / 출력 64k tokens
- **지식 컷오프**: 2025년 1월
- **지원 모달리티**: 텍스트 + 이미지 + 오디오 + 비디오 + PDF 입력 / 텍스트 출력

### 2.2. 2.5 Flash 대비 결정적 차이

**(a) Reasoning Native**
Gemini 3 는 **"reasoning model"** 이다. 즉:
- 응답 전에 내부 사고(thinking) 단계를 **자동 수행**
- SWE-bench Verified 78% (Gemini 2.5 Pro 수준)
- 2.5 Pro 대비 3배 빠른 추론, 비용 분수

**시사점**: "체인 오브 쏘트" 를 프롬프트로 구현할 필요 없음. 모델이 이미 한다. 프롬프트는 **결과만** 지시.

**(b) 덜 장황 (Less Verbose)**
> "By default, Gemini 3 is less verbose and prefers providing direct, efficient answers."

**시사점**:
- "짧게 답해" 지시 불필요 — 기본값이 이미 짧음
- "대화 톤 캐릭터" 하려면 **명시적으로 "더 말하라"** 지시 필요
- 루나 같이 수다스러운 캐릭터엔 **"친한 언니처럼 카톡 2~4 말풍선으로 반응"** 등 **구체적 지시 필수**

**(c) 페르소나 고수 (Strong Persona Adherence)**
> "The model is designed to treat the persona it is assigned seriously and will sometimes ignore instructions in order to maintain adherence to the described persona."

**시사점**:
- 루나 페르소나를 **강하고 분명하게** 설정하면 모델이 알아서 따름
- 하지만 **페르소나와 지시가 충돌**하면 페르소나 우선 — 설계 시 **충돌 최소화** 필요
- 예: "루나는 장난스러워" + "매번 물음표 금지" → 충돌 시 루나의 "장난스러움" 이 이길 수 있음
- 해결: **"루나는 장난스럽지만 상대가 지칠 정도로는 캐묻지 않아"** 같이 **페르소나 자체에 제약 녹이기**

**(d) 과도한 프롬프트 엔지니어링 회피**
> "It may over-analyze verbose or overly complex prompt engineering techniques used for older models."

**시사점**:
- 길고 세세한 프롬프트 = **독**
- "5단계로 나눠서 생각해" 같은 **명시적 CoT 지시 금지**
- 체크리스트형 지시 (✅/❌) 는 약함 — Gemini 2.5 에선 효과적이었으나 3 에선 과잉
- **간결한 서술형** + **명확한 역할/목표** 가 최강

**(e) 기본 온도 1.0 권장**
> "For Gemini 3, we strongly recommend keeping the temperature parameter at its default value of 1.0."

**시사점**:
- 기존에 0.7~0.9 로 튜닝하던 습관 버려라
- 1.0 이 gemini 3 의 **reasoning 최적화 기준점**
- 낮추면 오히려 **loop, degrade** 발생 가능성
- 루나 응답의 자연스러움도 1.0 에서 가장 좋음 (검증됨)

### 2.3. thinking_level 파라미터 (신규)

Gemini 3 의 **가장 중요한 신 파라미터**. 기존 `thinking_budget` 을 대체:

- `minimal` — 거의 thinking 안 함. 즉답.
- `low` — 가벼운 내부 사고.
- `medium` — 기본값. 중간 수준 reasoning.
- `high` — 최대 thinking. 복잡한 논리.

**루나 우뇌에 적합한 레벨**:

| 턴 유형 | 권장 레벨 | 이유 |
|---------|----------|------|
| 일상 톡, 간단 공감 | `minimal` 또는 `low` | 과한 thinking 은 대화 자연스러움 해침 |
| 표면-실제 mismatch 감지, 양가감정 | `medium` | 뉘앙스 판단 필요 |
| 위기 대응, 복잡한 자책, 사르카즘 | `high` | 안전 최우선, 정확 판단 필요 |
| 재분석 (REQUEST_REANALYSIS) | `high` | 이미 한 번 어긋났으므로 |

**동적 제어 전략**:
- 좌뇌 `complexity` 1~2 → `minimal`
- 좌뇌 `complexity` 3 → `low`
- 좌뇌 `complexity` 4 → `medium`
- 좌뇌 `complexity` 5 또는 `crisis_risk=true` → `high`

이는 **단가와 속도** 최적화에 결정적. 90% 턴이 `minimal`/`low` 면 비용/레이턴시 모두 2.5 Flash 와 비슷.

### 2.4. thinking 노출 제어

**⚠️ 치명적 주의 사항**: Gemini 3 는 기본적으로 내부 thinking 을 **스트림에 포함** 해서 반환할 수 있음.

설정 안 하면:
```
[thinking] 유저가 말 끊은 게 중요해 보임. "신..." 에서 멈춤...
[response] 아 너 방금 말 끊은 게 포인트야...
```
유저에게 thinking 부분이 **그대로 노출** 됨 → 치명적.

**필수 설정**:
```ts
thinkingConfig: {
  thinkingLevel: 'minimal' | 'low' | 'medium' | 'high',
  includeThoughts: false,  // 반드시!
}
```

현재 `src/lib/ai/provider-registry.ts` 의 Gemini 호출에 이 설정이 **없음**. Gemini 3 전환 시 **반드시** 추가해야 함.

### 2.5. 최적 프롬프트 구조 (Google 공식 권장)

Gemini 3 공식 문서의 권장 패턴:

```
Role + Goal + Constraints + Examples + Output format
```

**Role** — 누구인지. **짧고 구체적**.
- 좋음: "너는 29살 친한 언니 루나야."
- 나쁨: "너는 따뜻하고 공감적이고 가끔 장난스럽고..." (형용사 나열)

**Goal** — 뭘 할 건지. **행동 목표**.
- 좋음: "친구 연애 얘기에 반응해. 카톡처럼."
- 나쁨: "유저의 감정적 니즈를 파악하여 적절한 공감과..."

**Constraints** — 제약. **최소한만**.
- 좋음: "상담사 말투 쓰지 마."
- 나쁨: 20개 금지 항목 리스트

**Examples** — 예시. **1~2개**.
- 좋음: 루나 실제 대화 1개
- 나쁨: 10개 톤별 예시 × 15 카테고리

**Output format** — 출력. **엄격 X, 힌트 O**.
- 좋음: "말풍선 2-3개, ||| 로 구분."
- 나쁨: "정확히 3개 말풍선, 각 15자 이내..."

### 2.6. 현재 우뇌 프롬프트가 Gemini 3 관점에서 어긋나는 지점

현재 `ACE_V5_SYSTEM_PROMPT` (v75) 분석:

**✅ 잘된 부분:**
- "너는 루나야. 29살 친한 언니" — 명확한 Role
- "카톡으로 친구 연애 얘기 받고 있어" — 명확한 Goal
- "3단계 내면 독백" 메타포 — reasoning 모델에 자연

**❌ 과잉 부분:**
- "원칙 6개 불릿" — Gemini 3 는 이런 리스트에 덜 반응
- "절대 금지" 섹션 20여개 — Gemini 3 는 페르소나 강하면 알아서 안 함
- "🚨 v73 절대 금지 — 사고 노출" — Gemini 3 는 thinkingConfig 로 해결. 프롬프트에 중복.
- "루나 톤 라이브러리 15 카테고리" — 과잉. 2-3개 예시가 낫다.
- "LEFT_BRAIN_HINT 섹션" — 페르소나와 분리된 메타 지시 → 페르소나 강도 희석
- "REQUEST_REANALYSIS 섹션" — 같은 이유

**⚠️ 충돌 부분:**
- "얘가 직전에 네 말에 불만 표시했으면 새 주제 꺼내지 마" — 페르소나(상황 따라 융통성) 와 엄격 지시 충돌
- "매번 질문으로 끝내지 마" — "가끔 물어도 됨" 이라는 성격과 모순 가능
- 해결: **페르소나 안에 녹이기** — "루나는 취조는 싫어해. 자기 얘기 던지기도 좋아하고"

### 2.7. Gemini 3 페르소나 구축 패턴 (연구 기반)

**(a) 구체적 vs 추상적**
- 구체적: "29살. 홍대 근처 자취. 프리랜서 일러스트레이터. 연애 많이 해봄." ✅
- 추상적: "친근하고 공감적인 AI" ❌

**(b) 행동 기반 vs 형용사**
- 행동 기반: "친구가 바람폈다고 하면 같이 욕한다." ✅
- 형용사: "분노에 공감적임" ❌

**(c) 관계 정의**
- "너 앞의 사람은 친한 동생이야" — 상대 관계 **명시** → 자연스러운 반말/친밀감
- 없으면 Gemini 는 default polite 모드로

**(d) 내적 모순 허용**
- "루나는 장난스러운데 진지한 얘기 나오면 갑자기 차분해져" — 모순 허용 → 다차원 페르소나
- 단조로운 "항상 따뜻함" 보다 **입체적**

**(e) 부정적 특성 1~2개**
- "루나도 가끔 피곤함. 완벽한 상담사 아님." — 인간미
- 모델이 "완벽한 상담사" 흉내 내면 AI 티 남

### 2.8. few-shot 예시 설계

Gemini 3 는 **"3-5개 예시"** 가 golden zone.

루나 페르소나용 예시 구조:

```
### 루나의 실제 반응 예시

[예시 1 — 슬픔]
유저: 남친이랑 헤어졌어 ㅠㅠㅠ
루나: 아...|||진짜?|||언제??

[예시 2 — 분노 공명]
유저: 걔가 바람폈어
루나: 미쳤다 진짜|||나 듣는데도 열받는데

[예시 3 — 망상 재연]
유저: 여친이 사달라고 한 거 무시했어
루나: 아 그림 그려진다ㅋㅋ|||여친 옆에 와서 '오빠~' 했는데 너 폰만 봤지|||완전 패싱 당한 느낌일 듯
```

현재 v75 에는 15개 톤 카테고리 × 2~3 예시 = **30+ 예시**. 과잉. 3-5개로 축소 필요.

### 2.9. 입력 필드 설계 원칙 (Gemini 3 친화)

**좋은 입력:**
```
【유저가 방금 보낸 카톡】
"여친이랑 싸웠어"

【네 머릿속에 이미 떠오른 것들】
얘 지금 — 약간 답답함. 4턴째인데 여친 얘기만 맴돌고 있어.
너 최근 3턴 연속 질문으로 끝냈고 얘 반응이 짧아지는 중.
...
```

**나쁜 입력 (현재 v75 일부):**
```
### 좌뇌가 본 유저 상태
- 감정: 약한 부정 감정 (V=-0.3)
- 활성도: 보통 활성 (A=0.5)
- 통제감: 통제감 낮음 (D=0.3)

### derived_signals
- escalating: false
- suppression: true
```

**차이**: 전자는 **"네 내면에 떠오른 생각"** 으로 읽힘. 후자는 **"분석 보고서"**. Gemini 3 의 페르소나 엔진은 전자에서 루나로 **자연 동화**. 후자에선 "내가 루나를 연기하는 AI" 모드로.

### 2.10. Gemini 3 에게 절대 하면 안 되는 것

1. **"step by step" 또는 "let's think"** — reasoning 이미 내부에서 함. 외부 지시는 중복.
2. **"이런 감정을 느껴라"** 강요 — 페르소나 고수 특성 때문에 무시됨.
3. **JSON 과 자연어 출력 혼재 요구** — 하나만 골라라.
4. **긴 XML 구조 프롬프트** — Gemini 3 는 XML 을 Claude 만큼 선호 안 함. 마크다운이 더 자연.
5. **같은 지시 3번 반복** — 2.5 에선 효과 있었으나 3 에선 "잉, 왜 계속 같은 말?" 이라고 자체 판단.
6. **페르소나 + 분석 모드 혼합** — 하나 선택. 우뇌는 **페르소나 모드** 만.
7. **"너의 답변 후 반드시 검토해라"** — self-reflection 은 내부 thinking 에서 이미 함.

### 2.11. Gemini 3 의 "암묵 룰" (문서화 안 된 경험적)

- **첫 200자가 결정적**: 여기서 페르소나 못 잡으면 끝까지 애매함
- **마지막 100자 효과 큼**: 마무리에 "이제 너로서 반응해" 같은 **리마인더** 효과적
- **이모지 섹션 구분자는 OK**: `## 🫀` 같이 system prompt 내 **구조 시각화** 에는 도움. 단 "출력에 복사 X" 는 별도 명시.
- **대괄호 태그 선호**: `[XXX:...]` 형태는 JSON 보다 Gemini 3 친화적 (이미 많이 학습됨)
- **한국어 카톡 톤**: "~야", "~잖아", "ㅋㅋ", "ㅠㅠ" 자연스러움. "다움체" ("하시다", "말씀하시다") 는 오히려 어색.

### 2.12. 정리: Gemini 3 친화 프롬프트 설계 5원칙

1. **간결 + 명확** (체크리스트 X, 서술 O)
2. **페르소나 깊게 정의** (행동 + 관계 + 내적 모순)
3. **3-5개 예시** (많으면 분석 모드 진입)
4. **reasoning 은 모델에게 맡기기** ("생각해보라" X)
5. **thinking_level + includeThoughts=false 필수 설정**

이 5원칙이 섹션 5 (우뇌 아키텍처) 설계의 기반이 됨.
